# package.json 文件中配置项的作用
### 1. bin：定义命令行工具入口
作用：定义命令行工具的入口文件或命令别名。


### 2. main：定义模块的主入口
作用：定义模块的主入口文件.


### 3. module：指定 ESM 模块入口
作用：指定 ESM 模块的入口文件。

### 4. types：指定类型声明文件
作用：指定类型声明文件。

### 5. files：控制发布文件范围
作用：控制发布时包含的文件范围。

### 6. exports：高级模块导出控制
作用：精确控制包的导出路径，支持条件导出（按环境、模块类型等）。

### 7. scripts：定义构建脚本
作用：定义项目构建、测试、启动等脚本命令。

### 8. dependencies：依赖包
作用：定义项目运行时依赖的包。

### 9. devDependencies：开发依赖
作用：定义项目开发时依赖的包。

### 10. peerDependencies：peer 依赖
作用：定义与宿主环境共享的依赖包。

### 11. optionalDependencies：可选依赖
作用：定义可选安装的依赖包。

### 12. workspaces：工作区
作用：定义 monorepo 工作区。

### 13. publishConfig：发布配置
作用：定义发布时的配置项。

### 14. engines：引擎要求
作用：定义项目运行所需的引擎版本。

# turbo
turbo 是一个命令行工具，用于管理大型项目的构建、测试、部署等。

### 工作流程优势
1. 只能缓存： turbo 会缓存任务结果，避免重复构建
2. 并行执行：无依赖关系的任务可以并行执行
3. 增量构建：只构建有变化的任务
4. 依赖分析：自动分析任务依赖，优化执行顺序
5. Monorepo 支持： 特别适合多包项目的管理

# Monorepo
Monorepo 是一种代码管理策略，将多个相关的项目或包放在同一个代码仓库中进行管理。与传统的多仓库（每个项目一个独立仓库）相比，Monorepo 有以下好处：    
### 代码共享和服用
* 多个包共享公共代码、工具和配置
* 减少重复代码，提高开发效率

### 统一的依赖管理
* 所有包使用相同版本的依赖项
* 避免版本冲突和依赖地狱
* 通过工作区（workspaces）机制管理内部依赖

### 原子性提交
* 一次提交可以包含多个包的修改
* 确保相关变更的同步性
* 简化代码审查和回滚流程

### 统一的工具链
* 共享构建配置、测试配置、代码规范
* 统一的 CI/CD

## 缺点
#### 仓库体积增加
* 所有历史记录和代码都在一个仓库中
* 克隆和操作可能较慢

#### 权限管理负载
* 需要更精细的权限控制
* 不同团队可能需要不同的访问权限

#### 构建复杂度增减
* 需要更智能的构建系统来避免重复构建
* 依赖关系管理更复杂
